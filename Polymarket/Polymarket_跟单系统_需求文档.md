# Polymarket 跟单交易系统 — 产品需求文档 (PRD)

**版本:** v1.0 | **日期:** 2026年3月11日 | **状态:** 草稿

---

## 1. 项目概述

### 1.1 项目背景

Polymarket 是基于 Polygon 区块链的预测市场平台，所有交易数据链上公开透明。平台上存在少数高胜率的"大户"钱包，其交易行为具有显著的信号价值。本项目旨在构建一套完整的跟单交易系统，实现对目标钱包的监控、信号识别、虚拟验证和自动执行。

### 1.2 项目目标

- 实时监控目标大户钱包的交易行为，延迟控制在 5 秒以内
- 建立多维度的钱包评估体系，筛选真正有边际的跟单对象
- 通过虚拟交易系统验证策略有效性，再投入真实资金
- 支持从虚拟交易无缝切换到实盘自动执行
- 提供完整的风控、监控和绩效分析能力

### 1.3 技术栈选型

- **编程语言:** Python 3.11+
- **区块链交互:** web3.py + polymarket-apis (PyPI)
- **API 接入:** Polymarket CLOB API / Gamma API / Data API / WebSocket
- **数据存储:** PostgreSQL（交易记录）+ Redis（实时缓存）
- **消息通知:** Telegram Bot API
- **部署:** VPS（低延迟）+ Docker

## 2. 系统架构

### 2.1 整体架构

系统分为五层：数据采集层 → 信号处理层 → 策略引擎层 → 执行层 → 监控分析层。各层通过消息队列解耦，支持独立扩展和维护。

| 模块 | 功能描述 | 优先级 |
|------|----------|--------|
| 数据采集层 | WebSocket 实时订单簿 + 链上事件监听 + 定时拉取排行榜数据 | P0 - 核心 |
| 信号处理层 | 钱包评估、交易解析、信号生成、筛选过滤 | P0 - 核心 |
| 策略引擎层 | 跟单规则、仓位计算、风控检查、下单决策 | P0 - 核心 |
| 执行层 | 虚拟交易引擎 + 实盘下单引擎（可切换） | P0 - 核心 |
| 监控分析层 | 绩效报表、实时仪表盘、报警通知 | P1 - 重要 |

## 3. 详细功能需求

### 3.1 数据采集模块

#### 3.1.1 目标钱包监控

- 支持添加/删除/编辑监控钱包地址列表
- 通过 WebSocket 实时监听目标钱包的新交易事件
- 交易解析：识别买入/卖出方向、市场名称、价格、数量
- 支持监听 Polygon 链上 CTFExchange 合约事件作为备用数据源
- 延迟目标：从目标钱包交易确认到系统接收信号 < 5 秒

#### 3.1.2 市场数据采集

- 实时订单簿深度数据（买卖盘口、价差、流动性）
- 市场元数据（市场 ID、结算规则、到期时间、标签分类）
- 历史价格数据（多时间粒度：1h / 6h / 1d / 1w / 1m）
- 排行榜数据（每日定时拉取利润榜、交易量榜）

#### 3.1.3 数据存储

- PostgreSQL 表设计：watched_wallets、wallet_trades、markets、orderbook_snapshots、signals、virtual_trades、live_trades
- Redis 缓存：实时价格、订单簿快照、最新信号状态
- 数据保留策略：交易记录永久保存，订单簿快照保留 30 天

---

### 3.2 钱包评估与信号模块

#### 3.2.1 钱包筛选标准

系统应自动评估目标钱包的跟单价值，评估维度包括：

- **总盈亏 (PnL)：** 要求历史总盈利 > $50,000
- **胜率：** 已结算市场胜率 > 55%，样本量 > 50 笔
- **近期表现：** 近 30 天盈利为正，且近 7 天无大额亏损
- **领域专业度：** 按市场分类拆分胜率，只在其擅长领域跟单
- **交易频率：** 月均 < 200 笔（排除纯算法机器人）
- **流动性偏好：** 主要交易高流动性市场（日交易量 > $100K）

#### 3.2.2 信号生成规则

- **单钱包信号：** 目标钱包发生大额交易（> $5,000）时触发
- **篮子共识信号：** 同一领域 5-10 个钱包中超过 80% 买入同一方向时触发（强信号）
- **滑点检查：** 当前价格相比目标钱包入场价移动超过 10% 时，自动放弃该信号
- **流动性检查：** 目标市场买卖价差 > 5% 时，标记为高风险信号
- **信号强度评分：** 综合钱包质量、交易规模、篮子共识度、流动性等因子计算 0-100 分

---

### 3.3 策略引擎模块

#### 3.3.1 跟单规则配置

策略引擎支持灵活配置，核心参数包括：

- **仓位计算模式：** 比例跟单（目标钱包的 X%）/ 固定金额 / 分层比例
- **最大单笔仓位：** 占总资金的比例上限（默认 10%）
- **最大单市场敞口：** 单个市场最大持仓比例（默认 20%）
- **最大滑点效应：** 当当前市场价差超过阈值时拒绝下单
- **市场白名单/黑名单：** 按分类、流动性、到期时间过滤
- **信号强度阈值：** 仅执行评分超过阈值的信号（默认 60 分）

#### 3.3.2 风控规则

- 每日最大交易次数限制（默认 20 笔/日）
- 每日最大亏损限制（占总资金比例，默认 5%），触发后自动暂停交易
- 累计最大回撤限制（默认 15%），触发后系统全面暂停并报警
- 同一市场重复下单冷却期（默认 10 分钟）
- 异常检测：目标钱包短时间内大量对冲操作时自动拦截

### 3.4 虚拟交易系统（Paper Trading）

虚拟交易是整个系统的核心验证环节，在投入真实资金前先用模拟资金跑通全部流程。

#### 3.4.1 核心功能

- **虚拟账户：** 初始资金可配置（默认 $10,000 USDC）
- **模拟下单：** 接收信号后以当时实时价格模拟成交，记录完整交易日志
- **滑点模拟：** 根据订单簿深度模拟真实滑点（而非理想化成交）
- **持仓管理：** 实时追踪虚拟持仓、未实现盈亏、可用余额
- **市场结算：** 当市场结算时自动计算虚拟持仓的最终盈亏

#### 3.4.2 绩效评估指标

- 总收益率 / 年化收益率
- 胜率 / 盈亏比 (Profit Factor)
- 最大回撤 / 最大连续亏损笔数
- 平均每笔盈利 / 平均持仓时间
- Sharpe Ratio（如有足够数据点）
- 按钱包、按市场分类、按信号类型拆分绩效
- 与基准对比（如随机买入、买入市场共识方向等）

---

### 3.5 实盘执行模块

#### 3.5.1 实盘切换条件

当虚拟交易满足以下条件时，可考虑切换到实盘：

- 虚拟跑通超过 30 天
- 总交易笔数超过 100 笔
- 累计收益为正且胜率 > 55%
- 最大回撤 < 10%

#### 3.5.2 实盘功能

- 通过 Polymarket CLOB API 自动提交限价单
- EIP-712 签名下单，私钥加密存储（环境变量 / 硬件钱包）
- 下单前二次确认机制（可配置是否需要人工确认）
- 下单失败重试机制（最多 3 次，间隔递增）
- 支持紧急全部平仓按钮（通过 Telegram 或控制台）

---

### 3.6 监控与通知模块

#### 3.6.1 Telegram Bot 通知

- **实时推送：** 目标钱包新交易、信号触发、下单执行结果
- **每日摘要：** 当日交易统计、盈亏、持仓变化
- **风控报警：** 触发止损、异常检测、系统错误
- **快捷指令：** `/status`、`/pause`、`/resume`、`/positions`、`/pnl`、`/emergency_close`

#### 3.6.2 Web 仪表盘（P2 - 后续迭代）

- 持仓总览、实时盈亏曲线
- 监控钱包活动时间线
- 信号历史和命中率统计
- 策略参数在线调整界面

## 4. 核心数据模型

### 4.1 watched_wallets — 监控钱包表

| 字段 | 类型 | 说明 |
|------|------|------|
| wallet_address | VARCHAR(42) | 钱包地址（主键） |
| alias | VARCHAR(100) | 别名 / 标签 |
| category | VARCHAR(50) | 擅长领域（politics / crypto / sports...） |
| total_pnl | DECIMAL | 历史总盈亏 |
| win_rate | DECIMAL | 胜率 |
| recent_30d_pnl | DECIMAL | 近 30 天盈亏 |
| quality_score | INTEGER | 评分 0-100 |
| is_active | BOOLEAN | 是否激活监控 |
| updated_at | TIMESTAMP | 最后更新时间 |

### 4.2 signals — 信号记录表

| 字段 | 类型 | 说明 |
|------|------|------|
| signal_id | UUID | 信号唯一 ID |
| signal_type | ENUM | single_whale / basket_consensus |
| source_wallets | JSON | 触发信号的钱包列表 |
| market_id | VARCHAR | Polymarket 市场 ID |
| direction | ENUM | YES / NO |
| entry_price | DECIMAL | 目标钱包的入场价 |

| slippage_pct | DECIMAL | 与目标入场价的偏差百分比 |
| signal_score | INTEGER | 信号强度评分 0-100 |
| action_taken | ENUM | executed / skipped / pending |
| skip_reason | VARCHAR | 跳过原因（滑点过大 / 风控拦截 / ...） |
| created_at | TIMESTAMP | 信号生成时间 |

### 4.3 virtual_trades — 虚拟交易记录表

| 字段 | 类型 | 说明 |
|------|------|------|
| trade_id | UUID | 交易唯一 ID |
| signal_id | UUID | 关联信号 ID |
| market_id | VARCHAR | 市场 ID |
| direction | ENUM | BUY_YES / BUY_NO / SELL_YES / SELL_NO |
| entry_price | DECIMAL | 模拟成交价（含滑点） |
| quantity | DECIMAL | 份额数量 |
| cost | DECIMAL | 总成本 |
| exit_price | DECIMAL | 平仓 / 结算价格 |
| pnl | DECIMAL | 本笔盈亏 |
| status | ENUM | open / closed / settled |
| created_at | TIMESTAMP | 建仓时间 |
| closed_at | TIMESTAMP | 平仓 / 结算时间 |

## 6. 风险提示

- 跟单交易本质上依赖他人判断，大户策略可能随时衰减或变化
- 市场流动性不足时，跟单者可能成为大户的"退出流动性"
- 滑点和时延会显著侵蚀策略收益，特别是在快速变化的市场中
- 链上数据虽然公开，但大户可能使用多钱包隐藏真实仓位
- 建议始终先用虚拟交易充分验证，再以小资金试水实盘
- 注意所在地区的法律合规要求
